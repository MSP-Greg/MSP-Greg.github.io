Update 2016-07-16

  • Class 'Direct Subclasses' list now appears with the information at the top of the page.
  • Aliases are now shown in both the 'Methods List' and the class and module pages.
  • Aliases that are defined on attributes are now shown in the 'Attribute Summary' and 'Attribute Details' sections.
  • Attributes that have distinct read and write source code are now shown separately, and listed in the 'Methods List'. If the attribute is only defined by attr_accessor, only the 'read' attribute is listed.

General Notes

  • In Ruby, and to some degree, in Rails, the documentation is formatted for generation with RDoc. I have tried to make use of as much of the RDoc information as possible, but there may be info included with RDoc that is not shown in these docs. At present, yard-t2 does not parse the RDoc :stopdoc: directive, so some classes, modules, and methods may be shown that are not shown in RDoc. Comments to Gitter.
  • YARD and yard-t2 normally date stamp the footer of all pages. To minimize changes on the GitHub site, the date stamp has been removed.
  • Generally, as versions progress, documentation becomes better and also better formatted.
  • This documentation is available on GitHub, but since it uses XMLHttpRequest, it must be served by a web server.

Ruby Notes

  • Since Ruby 'Core' has quite a few Exception classes defined as root objects, they are listed in a separate 'Exceptions List'.
  • Please see the documents listed in the 'Docs List'.
  • None of the libraries contain the tk extension.
  • Ruby source is included in all, c source is only included in Ruby 2.4.
  • 'module function' instance methods are not shown, but the class methods are noted as such.
  • Alias documentation is handled differently, depending on whether the source is Ruby or c. Fix later.

Ruby on Rails Notes

  • The default Rails .yardopts file does not show private or protected objects. If you would like that changed, please comment.
  • The 'Rails Guides' are listed in the 'Docs List'.
  • The current version that is documented is listed in the Rails::VERSION module.

UI Info

UI Layout & General

On a wide screen desktop monitor, the layout is composed of three main panes (left to right):

  • the LIST pane, which shows the API doc lists (class / module list, methods list, documents list, etc)
  • the 'Table of Contents' (TOC) pane
  • the content pane, which includes the document and the main header

The two panes can each be left docked, hidden, or floated individually on the right side.

Browser window width determines the layout.

  • Narrow - will not allow the LIST or TOC panes to be left anchored
  • Medium - will allow only one pane to be left anchored
  • Wide - allows both panes to be left anchored


Pane Control

The upper right of the header has four buttons for controlling the LIST and TOC panes visibility and location, along with a global expand/collapse for source code.

Standard Key Assignments

If the LIST pane is visible and it is showing the method list, hitting any key will move the list to the top 'first character' match.

Hitting escape will empty the LIST pane search input box, resetting the LIST pane to the current list. Note that this will not reload the list from the server.

Alt-Control Key Assignments

Key Result

d Shows and loads the 'Docs List',  will show the LIST pane if it's not visible

List Search

Search is not case sensitive. If the first letter of the search string is upper case, the search will be a 'start of string' search, otherwise it matches anywhere. Thought was given to other methods, but in deference to tablet use, using uppercase as the switch seemed appropriate. The escape key will reset the search, as will the 'purple cross' that appears on the search icon.

When searching the 'Method List', one can also prefix the search string with a period (for class methods) or a # (for instance methods).

Suggestions, bugs, etc


yard-t2 - General Info

yard-t2 is a plug-in for YARD (created by Loren Segal), a Ruby documentation system. yard-t2 loads pages using XMLHttpRequest, so the content it generates must be served via a web server.

I will be releasing it as a Ruby gem, but more work is needed refactoring code, generating RSpec tests, etc.
